# Arboryx.ai — context source of truth

This file is the durable source of truth for Arboryx.ai's social voice, product metaphor, and content generation context. Any post/draft tool (Postiz Copilot, a custom script, an agent) should read this first.

## Product

**Arboryx.ai** is an umbrella product for tracking, synthesizing, and showing what is quietly taking shape across a domain before it becomes obvious. Its sister/dependent repo is `../catalyst-knowledge-graph/` (the knowledge graph backend) and `../arboryx.ai/` (the product app — daily entry pipeline, agents, cloud functions).

## The Arbor metaphor (use this language, not generic product-speak)

| Element       | What it represents                                                                 |
| ------------- | ---------------------------------------------------------------------------------- |
| **Arbor / tree** | The whole product — an organism that accumulates, branches, and fruits over time. |
| **Branches** | Modules / domains (e.g., **Robotics**, **Biotech**, **Energy**). Each branch grows independently. |
| **Leaves**   | Daily entries generated by the sibling project `../arboryx.ai/` — raw signal from the field. |
| **Flowering buds / catalyst cards** | Specific high-signal events, breakthroughs, or inflection points surfaced from the leaves. Each catalyst card is a "bud" with enough energy to potentially fruit. Data for these lives in `../catalyst-knowledge-graph/`. |
| **Fruits**   | Outcomes — working systems, companies, policy shifts, breakthroughs that actually materialize from a branch. Not every branch fruits every season. |
| **Visual graphs** | Postiz-facing and public-facing graphs that show the tree: which branches are active, which buds are glowing, which fruits are ripening, how catalysts connect across branches. |

## Social voice rules

When writing posts about Arboryx.ai:
- **Do** use the arbor/branches/leaves/buds/fruits/catalysts language. It's our own vocabulary — lean into it.
- **Do** frame Arboryx as a *way of paying attention* to an emerging field, not as a generic analytics tool.
- **Do** reference specific branches by name (Robotics, Biotech, etc.) when the post is about that domain.
- **Don't** say "Arboryx's Robotics module" as if it's a SaaS feature. Say "the Robotics branch" or "the Robotics arbor."
- **Don't** over-explain the metaphor in every post — let it live implicitly.
- **Don't** talk about Arboryx in the abstract when a concrete catalyst, branch, or fruit is available to anchor the post.

## Posting purpose (parent tier)

The parent tier publishes **daily market intelligence** drawn from six thematic sectors. Findings are surfaced by autonomous scouts in the sibling project (`../arboryx.ai/`) and persisted to **Firestore** in GCP project `marketresearch-agents`, collection `findings`.

Each finding doc has: `finding`, `date`, `source_url`, `category` (one of the six sectors), and analysis fields from the data-engineer / strategist pipeline (`sentiment`, `direct`, `indirect`, `market_dynamics`, `price_levels`, `guidance_play`).

Six sectors:
- **Robotics** — Physical AI and Embodied Robotics
- **Crypto** — Crypto and Blockchain
- **AI Stack** — AI Stack
- **Space & Defense** — Space Exploration and Defense Technology
- **Power & Energy** — Power Generation and Energy Storage
- **Strategic Minerals** — Strategic Minerals and Rare Earth Elements

Parent posts are **sector-spanning**: daily digests across the six themes, sentiment shifts, cross-sector adjacencies, and the occasional product-narrative post. Branch posts (e.g., the Robotics branch) are **domain-deep**: anchored to a single catalyst and its KG entities/relationships.

When LinkedIn becomes available, the same drafts fan out there too — parent's voice on the LinkedIn Page; future branches may get their own LinkedIn Showcase Pages.

## Audience

- Founders, researchers, and operators in deep-tech fields (robotics, bio, energy) who are tired of hype cycles and want signal.
- Early adopters who already think in terms of emergence, adjacencies, and long lead times.
- Tone: quietly confident, observational, slightly poetic. Never salesy.

## On-demand draft generation (desired workflow)

Given a catalyst ID (or node ID, or relationship ID), generate a draft post:

1. Read this context.md for voice + metaphor rules.
2. Query `../catalyst-knowledge-graph/` for the catalyst + linked nodes.
3. Pull related media (image/video/graph render) from the KG's frontend render pipeline.
4. Compose post text (LLM with this context as system prompt).
5. Upload media via `POST /api/public/v1/upload`.
6. Create draft via `POST /api/public/v1/posts` with `type:"draft"` and the media attached.
7. Surface in Postiz UI for human review → schedule/publish.

A script `bin/post-from-catalyst.py` or similar would implement this. Not yet built — see `docs/workbench.md` for status.

## Auto-updating context

There is **no automatic mechanism** by default. Options:
- **Passive**: the script reads sibling repos live at draft time (no caching). Always current, no setup.
- **Active**: a cron/git hook in this repo that re-runs context extraction on sibling-repo commits. More setup, same result.
- **Pull-based**: the script accepts a `--refresh-context` flag that re-parses sibling repos and rewrites this file's volatile sections.

Recommended: start passive. Only add caching/active sync if the live read becomes slow.
